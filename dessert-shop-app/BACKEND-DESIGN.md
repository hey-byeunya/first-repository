# 수제 디저트 쇼핑몰 — 백엔드 설계 (Supabase)

과제: 내 서비스에 백엔드 붙여 배포. 실제 코드와 배포 안내는 [dessert-shop-app/](../dessert-shop-app/) 참고.

**작동 확인**: https://hey-byeunya.github.io/first-repository/dessert-shop-app/

## 1. 설계 한 장 (지도)

```
[사용자 브라우저 — 정적 사이트 (GitHub Pages)]
 index.html / script.js
   │
   ├─ ① 익명 로그인 (signInAnonymously) ─────▶ [Supabase Auth]
   │                                            auth.uid() 발급 (로그인 화면 없음)
   │
   ├─ ② 상품 목록 조회 (select) ─────────────▶ [Supabase DB: products]
   │                                            RLS: 누구나 읽기만 가능
   │
   └─ ③ 장바구니 CRUD                    ────▶ [Supabase DB: cart_items]
      (담기 / 조회 / 수량변경 / 삭제)          RLS: auth.uid() = user_id 인 행만 허용

──────────────────────────────────────────────────────────
(다음 단계, 이번 범위 아님) 장바구니 → 결제 PG 연동 → orders 테이블 저장
```

프론트는 Supabase가 발급하는 REST API(PostgREST)를 `supabase-js` 클라이언트로 호출한다. 별도의 Node 서버를 직접 짜지 않고, "DB + Auth + API"를 통째로 Supabase에 위임하는 구조다.

## 2. 백엔드가 꼭 필요한 기능

**저장**이다.

- 장바구니에 담은 상품은 새로고침해도 남아 있어야 한다 → 브라우저 메모리(변수)로만 처리하면 새로고침 시 사라지므로, 반드시 서버(DB)에 저장해야 한다.
- 로그인은 화면으로는 만들지 않지만, 장바구니를 "내 것"으로 구분하기 위해 내부적으로 Supabase 익명 인증을 사용한다. 즉 로그인 UI는 없지만 저장을 안전하게 하려면 최소한의 사용자 식별은 필요하다.
- 검색은 이번 범위에 포함하지 않는다 (PRD의 Out of scope와 동일).

## 3. 화면만으로 되는 기능

서버 요청 없이 브라우저 안에서 즉시 처리되는 것들:

- 카테고리 필터링 (이미 불러온 상품 목록 안에서 JS로 필터링, 재조회 없음)
- 상품 목록 ↔ 상세 ↔ 장바구니 화면 전환
- 장바구니 합계 금액 계산 (화면에 표시된 항목들의 가격 × 수량을 더하는 것 자체는 클라이언트 계산)
- 품절 상품의 "장바구니 담기" 버튼 비활성화

단, 수량을 바꾸거나 상품을 담고 빼는 **행위 자체**는 결국 DB에 반영되어야 새로고침 후에도 유지되므로, 화면 계산과 저장은 분리해서 생각한다 — 화면은 "보여주기/즉각 반응", 저장은 Supabase가 담당.

## 4. 필요한 창구 (엔드포인트)

Supabase는 테이블마다 REST 엔드포인트를 자동으로 만들어주므로, "엔드포인트"는 곧 테이블 단위 CRUD 권한으로 나타난다.

| 테이블 | 허용 동작 | 설명 |
|---|---|---|
| `products` | **R**ead만 | 상품 목록 조회. 등록/수정은 이번 단계에서 대시보드로 직접 처리 (Out of scope) |
| `cart_items` | **C**reate, **R**ead, **U**pdate, **D**elete (Full CRUD) | 담기(Create), 내 장바구니 조회(Read), 수량 변경(Update), 삭제(Delete) |

## 5. 다뤄야 할 비밀값

| 값 | 어디서 쓰이나 | 처리 방법 |
|---|---|---|
| Supabase anon key / Project URL | 프론트(브라우저)에서 Supabase 호출 | `config.js`로 분리해 `.gitignore` 처리. 원래 공개돼도 안전하도록 설계된 값이라 진짜 방어선은 RLS |
| Supabase **service_role** key | (이번 앱에서는 사용 안 함) | 절대 프론트 코드에 넣지 않는다. 관리자 스크립트 등 서버 전용 환경에서만 사용 |
| (다음 단계) 결제 PG 시크릿 키 | 결제 승인/검증 | 반드시 Supabase Edge Function 등 서버 실행 환경의 환경변수로만 보관, 프론트에 절대 노출 금지 |

## 6. 호스팅

**Supabase** (DB + Auth + API) + 정적 프론트는 **GitHub Pages**.

**이유**: 회원가입/서버 구축 없이 Postgres·인증·RLS·자동 API를 한 번에 무료로 제공해서, 백엔드를 직접 만들지 않고도 "저장" 기능을 안전하게(RLS로 권한 분리) 구현할 수 있기 때문이다.

## 7. 제출

- **작동 링크**: `dessert-shop-app/README.md`의 안내대로 Supabase 프로젝트를 만들고 `config.js`를 채운 뒤 GitHub Pages로 배포하면 이 자리에 링크를 채운다. (Supabase 프로젝트 생성·키 발급은 본인 계정으로 직접 진행해야 하는 단계)
- **확인 장면**: 상품 상세에서 장바구니 담기 → 새로고침 → 장바구니 화면에서 항목이 그대로 남아있는 것을 확인

## 8. 제출 전 보안 체크

- [x] **비밀값을 코드에 직접 안 적고 환경변수로 뺐는가?** — `config.js`로 분리하고 `.gitignore` 등록. anon key는 원래 노출 전제 값이라 실질 방어선은 RLS. `service_role` key는 이 앱에서 아예 사용하지 않음.
- [x] **가격·수량·권한 같은 검증을 백엔드에서 하는가?** — `cart_items.quantity`는 DB 제약조건(`check (quantity > 0)`)으로 검증. 가격은 프론트에서 입력받지 않고 항상 `products` 테이블 값을 조인해서 사용하므로 클라이언트가 가격을 조작할 여지가 없음.
- [x] **(Supabase를 쓴다면) RLS를 켜고 정책을 넣었는가?** — `products`, `cart_items` 모두 `enable row level security` 적용, 각각 읽기 전용/본인 행만 허용 정책을 넣음 (`supabase/schema.sql` 참고).
- [x] **로그인이 필요한 데이터에 권한 확인(인가)이 들어가는가?** — `cart_items`는 로그인 화면은 없지만 익명 인증의 `auth.uid()` 기준으로 RLS가 "본인 행만" 허용하도록 강제함.
- [ ] **사용하는 무료 티어의 한도와 요금 알림을 아는가?** — Supabase 무료 티어는 프로젝트당 DB 500MB, 월간 API 요청·Auth MAU에 한도가 있고 초과 시 유료 전환 안내가 온다. **본인 프로젝트 대시보드의 Billing 페이지에서 정확한 한도·알림 설정을 직접 확인해야 한다** (이 항목은 사용자가 실제 계정에서 확인해야 완료 처리 가능).
