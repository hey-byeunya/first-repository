# 수제 디저트 온라인 가게 — 서비스 기획서 (PRD)

## 무엇을

수제 디저트를 파는 온라인 가게. **선물 사는 사람이 편하게 고를 수 있도록** 돕는 것이 이 서비스의 핵심 목표다.

## 누구를 위해

관계나 기념일 구분 없이, 언제든 "디저트 선물 하나 사야 하는" 사람 누구나. 특정 시즌(발렌타인 등)이나 특정 관계(부모님용 등)에 맞추지 않고, 상황과 무관하게 빠르게 고를 수 있는 범용 선물 서비스를 지향한다.

## 공급 구조

단일 사장(1인 브랜드)이 직접 만드는 디저트만 판매한다. 여러 입점사를 모으는 마켓플레이스가 아니다.

## 핵심 기능

### 1. 상품 탐색 — 큐레이션 세트

- 낱개 상품을 나열하지 않고, 미리 구성된 "선물 세트" 단위로 보여준다.
- **가격대 기준**으로 세트를 분류한다 (예: 1~2만원대 / 3~5만원대 / 5만원 이상).
- 구매자는 가격대만 고르면 그 안의 세트 중 하나를 선택하는 것으로 구매를 끝낼 수 있어야 한다.

### 2. 선물 부가 기능

- **포장**: 선물 포장 옵션 제공 (디자인 선택 가능).
- **메시지 카드**: 구매자가 직접 문구를 입력해 카드에 담을 수 있다.
- 배송일 지정 기능은 V1 범위에서 제외한다 (향후 버전 검토 대상).

### 3. 수령자 정보 입력

두 가지 방식을 모두 지원한다.
- **직접 입력**: 구매자가 받는 사람의 주소를 바로 입력 (일반 쇼핑몰 방식).
- **선물하기 링크**: 구매자는 결제만 하고, 링크(카톡/문자 등으로 전달)를 받은 수령자가 직접 자신의 주소를 입력. 구매자가 주소를 몰라도 선물 가능.

### 4. 결제

자체 결제 연동(카드, 간편결제)을 통해 사이트 안에서 결제가 완료된다. PG사(예: 포트원/토스페이먼츠 등) 선정은 별도 결정 필요.

### 5. 배송

전국 택배 배송. 신선도 유지를 위해 상온/냉장 보냉 포장을 사용한다.

### 6. 생산량(재고) 관리

- 수제 특성상 상품/세트별로 **날짜별 판매 가능 수량**을 설정한다.
- 해당 날짜 수량이 소진되면 **품절 표시** 후 그 날짜의 주문을 막는다.
- 구매자는 허탕 없이 판매 가능 여부를 바로 확인할 수 있어야 한다.

### 7. 후기

텍스트와 사진을 함께 남길 수 있는 후기 기능을 제공한다. 상품 상세 페이지에 노출되어 구매 결정("선물 실패하면 안 된다"는 부담)을 돕는다.

### 8. 운영자(사장님) 관리 페이지

V1에서는 최소한의 관리 기능만 지원한다.
- 주문 목록 조회
- 주문 상태 변경 (접수 → 제작중 → 발송완료)
- 품절 처리

재고/매출 통계 등 고급 관리자 도구는 V1 범위에서 제외한다.

## 플랫폼

모바일 웹 중심의 반응형 웹사이트. 별도 앱 없이 브라우저에서 바로 이용 가능해야 하며, 특히 "선물하기 링크"를 받은 사람이 모바일에서 불편함 없이 주소를 입력할 수 있어야 한다.

## 화면 흐름 개요

1. 홈 — 가격대별 큐레이션 세트 목록
2. 상품 상세 — 세트 구성, 후기(텍스트+사진)
3. 주문/결제 — 포장 선택, 메시지 카드 작성, 수령자 정보 입력 방식(직접 입력 / 선물하기 링크) 선택, 결제
4. 선물하기 링크 수신 화면 — 수령자가 배송 주소 입력
5. 주문 완료
6. (운영자) 관리 페이지 — 주문 목록 / 상태 변경 / 품절 처리

## 예외 상황

| 상황 | 동작 |
|---|---|
| 주문하려는 날짜의 수량이 이미 소진됨 | 품절 표시, 해당 날짜 주문 불가 (다른 날짜/세트 안내) |
| 선물하기 링크를 받은 사람이 주소를 입력하지 않음 | 일정 기간(예: 7일) 후 링크 만료 처리, 구매자에게 안내 |
| 결제 후 취소/환불 요청 | V1은 관리자 페이지에서 수동 처리 (자동 환불 로직은 후순위) |
| 냉장 배송 중 파손/변질 클레임 | 관리자가 개별 대응 (교환/환불), 별도 정책 문서화 필요 |
| 메시지 카드 문구에 부적절한 내용 입력 | V1은 별도 필터링 없음 (향후 검토) |

## 향후 확장 계획 (V2 이후 — 백엔드 확장 고려)

V1은 회원가입 없이 **비회원(게스트) 주문** 방식으로 운영한다. 다만 아래 기능을 나중에 무리 없이 얹을 수 있도록 설계 단계에서 고려한다.

- **로그인/회원가입**: 재구매 편의, 주문 내역 조회, 위시리스트. V1 주문 데이터에 이메일/전화번호를 식별자로 남겨두면, 이후 이 값으로 회원 계정과 연결하기 쉽다.
- **정보 저장(마이페이지)**: 자주 쓰는 수령자 주소, 결제 수단 저장.
- **결제 고도화**: 적립금, 쿠폰, 정기구독(정기 선물) 등. V1의 "자체 결제 연동"이 이 확장을 전제로 한 선택이다.
- 이런 확장을 염두에 두고, DB 스키마 설계 시 `user_id`를 처음부터 nullable로 열어두는 등 계정 시스템을 나중에 붙이기 쉬운 구조를 권장한다.

## 참고 — 기술 스택 방향

별도 결정 사항이지만 논의된 방향을 기록해둔다.
- 커스텀 로직(선물하기 링크, 날짜별 재고)이 많아 기성 플랫폼(카페24/아임웹)보다 **직접 구현**이 유리하다고 판단.
- 관리자 페이지 개발 부담을 줄이려면 **Django**(Django Admin 기본 제공), 프론트 UX·최신 배포 편의성을 중시하면 **Next.js**를 검토.
