# 용어사전 작성 스타일 가이드

이 가이드는 용어사전 각 항목을 **어떤 톤과 구조로 쓸지** 정한다. 목표는 "코딩 입문자가 읽고 바로 이해하는, 비유가 살아있는 한국어 사전"이다.

## 원칙

1. **한국어로, 쉽게.** 전문 용어를 설명하는 문서지만, 설명 자체는 쉬워야 한다. 어려운 말을 또 다른 어려운 말로 풀지 않는다.
2. **비유가 핵심이다.** 거의 모든 항목에 일상 비유를 하나씩 넣는다. 좋은 비유는 정의보다 오래 기억된다. 이 대화에서 검증된 비유들 — 터미널/셸/커널 = 식당(창구·웨이터·주방장), API = 식당 창구, DOM = 인형극 무대, 해시 = 믹서기, JWT = 놀이공원 팔찌, 추상화 = 자동차 운전/리모컨 — 처럼 구체적이고 친근한 걸 고른다.
3. **왜 필요한지 먼저.** 많은 용어는 "이런 문제가 있어서 나왔다"는 맥락이 있을 때 이해된다. 정의만 던지지 말고, 필요하면 문제 상황을 짧게 깔아준다.
4. **정확성.** 약어 풀이, 수치, 동작 순서는 지어내지 않는다. 웹 용어는 MDN 한국어 문서로 확인한다. 헷갈리면 확인 후 쓴다.
5. **연결.** 관련 있는 다른 용어는 `related` 필드에 넣는다. 본문 문장 중간에서 자연스럽게 언급할 땐 `[[용어]]`로도 이을 수 있다. 사전 전체가 하나의 지도처럼 엮이는 게 이 문서의 강점이다.

## 항목 구조 (JSON 필드)

데이터는 마크다운이 아니라 JSON 객체로 등록한다. 필드 이름과 스키마 전체는 [SKILL.md](../SKILL.md)의 "데이터 스키마" 참고. 톤을 정할 때는 각 필드를 이렇게 채운다:

```
term        <용어>
original    <원어·약어 풀이>               (선택)
definition  한 줄 정의 — 무엇인지 한 문장. 비유를 살짝 섞어도 좋다.
analogy     {emoji, text} — 일상에서 바로 떠올릴 이미지. 이모지 하나 정도.
details     필요할 때만. 표, 4~6줄 코드, 추가 문단으로 보강.
related     [a, b]                         (다른 term 이름)
references  [{text, url}]                  (선택, 주로 MDN)
```

- 모든 필드를 억지로 채우지 않는다. 간단한 용어는 `definition` + `analogy` + `related`만으로 충분하다.
- `details`의 표는 "A vs B" 비교나 "종류 정리"에 특히 좋다.
- 코드는 짧게. 개념을 보여줄 최소한만.

## 완성 예시 (이 톤을 목표로)

---

```json
{
  "term": "DOM",
  "original": "Document Object Model",
  "group": "웹 기초",
  "definition": "브라우저가 HTML 문서를 프로그램이 다룰 수 있게 바꿔놓은 \"나무(트리) 구조\".",
  "analogy": {
    "emoji": "🎭",
    "text": "웹페이지가 인형극 무대라면, HTML은 대본, DOM은 무대에 실제로 세워진 인형들(조종 가능), JavaScript는 인형을 움직이는 조종사다. 대본만으론 인형이 안 움직인다."
  },
  "details": [
    {"type": "paragraph", "text": "HTML은 그냥 글자다. 브라우저가 이걸 읽어 부모-자식 관계의 트리로 정리하는데, 그게 DOM이다. 이 덕분에 JavaScript로 페이지를 실시간으로 바꿀 수 있다."},
    {"type": "code", "text": "document.querySelector(\"h1\").textContent = \"반가워요!\"; // 화면 글자가 바뀜"}
  ],
  "related": ["HTML", "CSS", "JavaScript"],
  "references": [{"text": "MDN — DOM 소개", "url": "https://developer.mozilla.org/ko/docs/Web/API/Document_Object_Model/Introduction"}]
}
```

```json
{
  "term": "해시",
  "original": "Hash",
  "group": "데이터·보안",
  "definition": "어떤 데이터든 넣으면 정해진 길이의 고유한 값으로 바꿔주는 것. 데이터의 \"지문\".",
  "analogy": {
    "emoji": "🥤",
    "text": "믹서기. 과일(원본)을 넣으면 주스(해시값)가 나온다. 같은 과일은 항상 같은 주스가 되지만, 주스를 다시 과일로 되돌릴 수는 없다(단방향)."
  },
  "details": [
    {"type": "paragraph", "text": "입력이 조금만 달라도 결과는 완전히 달라지고, 입력 길이와 무관하게 결과 길이는 항상 같다. 되돌릴 수 없어서 비밀번호 저장·파일 검증에 쓴다."},
    {"type": "table", "rows": [
      ["특징", "내용"],
      ["단방향", "해시값 → 원본 복원 불가"],
      ["고정 길이", "입력이 1글자든 5GB든 결과 길이 동일"],
      ["결정성", "같은 입력 → 항상 같은 해시"]
    ]}
  ],
  "related": ["해시 테이블", "JWT", "인덱스"]
}
```

## 색인(맨 위 목록)

문서 맨 위에는 수록된 용어를 한눈에 보는 색인을 둔다. HTML에서는 각 용어 카드로 점프하는 링크 칩으로 만든다. 주제별로 묶으면 훑어보기 좋다.

예시 묶음:
- **터미널·셸 계열**: 터미널, 셸, 커널, 프롬프트, bin, PATH
- **웹 3형제**: HTML, CSS, JavaScript, TypeScript, DOM
- **네트워크**: LAN, WAN, 사설 IP, HTTP, API
- **데이터·보안**: 쿼리, 인덱스, PK, 해시, JWT, 세션, 쿠키, RLS, .env
- **AI 계열**: LLM, RLHF, 할루시네이션, 벤치마크, Agent, 하네스, MCP, 스킬, 프롬프트
