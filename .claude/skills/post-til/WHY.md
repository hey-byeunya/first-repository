# post-til 스킬은 왜 만들어졌나

이 문서는 `post-til` 스킬이 **어떤 불편에서 출발했고, 그 불편을 어떻게 없앴는지**를 기록한 것이다.

---

## 1. 배경 — 무슨 상황이었나

개발·CS·AI 용어를 하나씩 채팅으로 물어보며 공부하고 있었다.
GUI, npm, node, DOM, HTTP, API, 해시, JWT, LLM, MCP, 스킬… 하루에도 수십 개.

문제는 이렇게 대화로 배운 내용이 **그때만 이해되고 뒤돌아서면 사라진다**는 것이었다.
그래서 "질문–답변"을 블로그(`hey-byeunya.github.io`)에 기록해두고 싶었다.

그런데 막상 블로그에 올리려니, 매번 손이 많이 가는 일이 한둘이 아니었다.

---

## 2. 불편 — 무엇이 번거로웠나

### ① 흩어진 대화를 매번 다시 정리해야 함
채팅에 흩어진 용어 설명을 골라내서, 질문–답변 형식으로 다시 다듬는 일을 매번 수작업으로 해야 했다.

### ② Jekyll 포스트 형식을 매번 맞춰야 함
블로그는 Jekyll(Chirpy 테마)이라 규칙이 있다.
- 파일명: `_posts/YYYY-MM-DD-제목.md`
- front matter: `title / date / categories / tags`
- Chirpy 전용 문법: `> ...\n{: .prompt-tip }` 같은 강조 박스
이걸 매번 기억해서 맞추는 게 번거로웠다.

### ③ 로컬 빌드 환경이 안 맞음 (가장 큰 함정)
미리보기를 하려고 `bundle exec jekyll build` 를 돌리면 이런 에러가 났다.
```
Could not find 'bundler' (2.6.9) required by your Gemfile.lock
```
원인은 셸이 **시스템 Ruby 2.6**(bundler 1.17)을 쓰고 있었기 때문.
정작 필요한 **Homebrew Ruby 3.4.5**(bundler 2.6.9)는 PATH에 안 잡혀 있었다.

### ④ "미래 날짜" 함정
포스트 시간을 무심코 현재보다 뒤로 적었더니, Jekyll이 그 글들을 조용히 **건너뛰었다.**
```
Skipping: _posts/....md has a future date
```
GitHub Pages도 똑같이 건너뛰기 때문에, 블로그에 글이 **안 보이는데 원인을 모르는** 상황이 생겼다.

### ⑤ 푸시가 거절됨
게시하려고 `git push` 하니 원격에 다른 커밋이 있어서 거절됐다.
```
! [rejected] main -> main (fetch first)
```
`git pull --rebase` 후 다시 푸시해야 한다는 걸 그때그때 떠올려야 했다.

### ⑥ 공개 블로그라 실수가 무섭다
`hey-byeunya.github.io`는 공개 저장소다. 확인 없이 잘못 올라가면 되돌리기 번거롭고,
검토 안 된 글이 그대로 인터넷에 노출된다.

---

## 3. 해결 — 스킬이 각 불편을 어떻게 없앴나

`post-til` 스킬은 위 여섯 가지를 **한 번의 요청**("오늘 배운 거 블로그에 올려줘")으로 처리하도록 만들었다.

| 불편 | 스킬이 없앤 방법 |
|------|-----------------|
| ① 흩어진 대화 정리 | 현재 대화에서 질문한 용어를 자동으로 모아 Q&A 포스트로 변환. 제목도 `뒤돌아서면 까먹는 약어 & 용어 풀이: <용어>`로 고정 |
| ② Jekyll 형식 | Chirpy front matter·파일명·`.prompt-*` 박스 규칙을 SKILL.md에 명시 → 매번 자동 준수 |
| ③ 빌드 환경 | `scripts/preview.sh`가 **Homebrew Ruby를 자동으로 PATH 앞에** 세워 실행 → bundler 버전 에러 원천 차단 |
| ④ 미래 날짜 | date를 **현재 시각**(`date "+%Y-%m-%d %H:%M:%S %z"`)으로 넣도록 규칙화 → 미래 날짜가 될 수 없음 |
| ⑤ 푸시 거절 | 게시 명령에 `push \|\| (pull --rebase && push)` 패턴을 문서화 → 거절돼도 자동 복구 |
| ⑥ 공개 실수 | **"푸시 전 승인" 게이트를 절대 규칙으로** 박음. 로컬 미리보기 → 명시적 승인 → 그다음에만 커밋·푸시 |

### 핵심 한 줄
> 흩어진 대화를 → 형식 맞춘 블로그 글로 바꾸고 → 안전하게(승인 후) 게시하는 과정을,
> **매번 손으로 하던 걸 스킬 하나로 묶었다.**

---

## 4. 곁들여 정리된 것

이 스킬을 만드는 과정에서 환경 문제도 함께 해결했다.
- `~/.zshrc`에 Homebrew Ruby PATH를 추가 → 앞으로 새 터미널에서 `bundle exec jekyll serve`가 바로 됨.
- 관련 배경은 메모리 `blog-posting`에도 기록해, 다음 세션에서도 "푸시 전 승인" 규칙이 유지되게 했다.

---

*관련: `SKILL.md`(사용법), `scripts/preview.sh`(미리보기), 자매 스킬 `glossary-builder`(용어사전 문서화).*
